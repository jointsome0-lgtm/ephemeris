# TickTick visual-parity audit

_Generated 2026-06-06 by a multi-agent workflow: per-view screenshot diff (live TickTick vs our app) + adversarial verification of each gap. Intentional deviations and test-data noise were filtered out._

**22 confirmed findings** — by view: Tasks / Today 4, Calendar (month) 3, Eisenhower Matrix 3, Focus / Pomodoro 6, Habit 1, Search 5.

Almost every fix is a small, localized CSS/template change.


## HIGH (2)

### [Search] Search icon renders at huge intrinsic size (no CSS)
- **severity** high · **effort** small · **category** iconography
- **evidence** TickTick shows no oversized search glyph at all; ours renders a ~480px black magnifying-glass SVG that dominates the entire content area. The .search-ico span and its SVG have NO sizing rules in style.css, so the icon falls back to its intrinsic size. There is also no global 'svg { width: ... }' fallback.
- **fix** In app/static/style.css add a rule for the inline icon, e.g. '.search-ico svg { width: 16px; height: 16px; display: block; color: var(--faint); }' (and ideally a global default like 'svg { width: 1em; height: 1em; }'). The icon should sit inside the input bar, not float as a hero graphic.
- **verified** Confirmed genuine and actionable. search.html line 8 wraps an inline SVG in a `.search-ico` span, but style.css has NO `.search-ico svg` rule and no global `svg { width:... }` fallback (verified: only .rail-ico, .lb-ico, .head-btn, etc. svg rules exist). The SVG therefore renders at its intrinsic size — my-search.png shows a ~480px black magnifying-glass hero graphic dominating the content area, with the search input shrunk to the bottom. TickTick (tt-search.png) shows no such oversized glyph; its content area is clean (filter chips / small empty-state illustration only). This is a chrome/iconography defect, not a data difference, not an intentional deviation, and clearly perceptible — it makes the screen look broken. Fix is a one-line CSS rule sizing `.search-ico svg` to ~16px (plus optionally a global svg default), so effort is small. High severity since it dominates the entire view.

### [Search] Empty-state not vertically centered / mispositioned
- **severity** high · **effort** small · **category** layout
- **evidence** TickTick centers the empty illustration + 'Nothing found' text in the middle of the viewport. Ours pushes the 'Search everything / Find tasks and notes…' text to the very bottom of the page, below the oversized icon and the input. The .empty-state has padding:80px 20px but is not centered in the available content height.
- **fix** In app/templates/search.html group the icon + title + subtitle into the .empty-state block and in style.css give .empty-state a larger min-height (e.g. min-height:60vh) with justify-content:center so it centers within the content area like TickTick.
- **verified** Confirmed real and visibly broken (not data). In app/static/style.css there are NO rules for .search-bar, .search-ico, or .search-input (grep returns nothing), so the search SVG renders at its unconstrained intrinsic size — that is the giant black magnifying glass filling our screenshot — and the unstyled input plus the .empty-state ("Search everything / Find tasks and notes…") simply flow underneath it, dumping the text near the bottom of the viewport. .empty-state only has padding:80px 20px (line 557) with no vertical centering in the content height. TickTick instead shows a small illustration + "Nothing found" text neatly centered mid-viewport. Any user would perceive ours as a broken layout, not a TickTick-like empty state. The claimed symptom (empty-state mispositioned / not centered) is accurate; root cause also includes the oversized icon, so the proposed fix (min-height:60vh + justify-content:center on .empty-state) helps but should be paired with sizing .search-ico/.search-bar. Bumped from medium to high because the raw oversized icon makes the page look broken, not just slightly off. Effort small: a handful of CSS rules.

### [Calendar (month)] Weekday header row is hugely inflated
- **severity** medium · **effort** small · **category** layout
- **evidence** In TickTick the Sun..Sat weekday labels sit in a thin fixed-height strip (~26px) directly above the first week row. In ours there is a large empty band roughly a full cell-height tall between the weekday labels and the first row of day numbers, so the whole grid is shoved down and proportions look off.
- **fix** In app/static/style.css, .cal-month uses grid-auto-rows: 1fr which gives the .cmw-dow header row the same 1fr height as a full week. Give the DOW row a fixed height instead of letting it expand: set an explicit grid-template-rows where the first (header) row is auto/fixed (e.g. grid-template-rows: auto repeat(N, 1fr)) or stop the .cmw-dow cells from stretching by giving .cal-month-dow its own row. Simplest: change .cmw-dow to not participate in the 1fr auto-rows (e.g. wrap the DOW strip outside the .cal-month grid as a separate flex/grid row of fixed ~30px height), keeping only the week cells in the 1fr grid.
- **verified** Confirmed genuine layout gap, visible in both screenshots and the CSS. In TickTick the Sun..Sat weekday strip is ~26px tall, sitting directly above the first week's day numbers (labels y~71, numbers y~101). In ours the day numbers don't start until y~204 — a ~130px empty band, making the first week row effectively double-height and shoving the whole grid down. Root cause confirmed in app/static/style.css:509-510: .cal-month uses grid-auto-rows:1fr and .cal-month-dow uses display:contents, so the DOW header row is sized as a full 1fr week row. Proposed fix (explicit grid-template-rows: auto repeat(6,1fr) so the header row is auto-height) is correct. This is design/proportion, not data; not an intentional deviation. Severity downgraded from high to medium: clearly perceptible polish issue but the calendar remains functional and recognizable. Effort small — a one-line CSS change.
